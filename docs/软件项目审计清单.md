# 软件项目审计清单（通用版）

> **文档定位**：适用于任何软件项目的通用审计框架，覆盖工程、安全、合规、性能、可维护性等全维度。
>
> **使用方式**：
> - 新项目发布前：完成 P0 + P1 全部审计
> - 正式版（v1.0+）发布前：完成 P0 + P1 + P2 全部审计
> - 金融级/生产级认证：完成 P0 + P1 + P2 + P3 全部审计
>
> **版本**：v1.0
> **最后更新**：2026-07-21

## 二、P0 必做审计（发布前）

### 2.1 工程化审计

**目标**：验证代码符合工程规范，编译通过，测试齐全，文档完整。

**检查项**：
- [ ] 代码格式化检查通过（无 diff）
- [ ] 编译零错误零警告
- [ ] Lint 检查通过（clippy/eslint/stylelint 等）
- [ ] 单元测试 + 集成测试全部通过
- [ ] 文档测试通过（doctest）
- [ ] API 文档生成无警告
- [ ] CI Pipeline 全部 gate 通过

**工具**：
- Rust: `cargo fmt --check` / `cargo clippy --all-targets` / `cargo test` / `cargo doc`
- Node.js: `eslint` / `prettier` / `jest` / `typedoc`
- Python: `black` / `ruff` / `pytest` / `sphinx`
- Go: `gofmt` / `golangci-lint` / `go test` / `godoc`

**通过标准**：所有 gate 100% 通过，0 警告 0 错误。

---

### 2.2 安全审计

**目标**：检测已知漏洞、禁用依赖、许可证合规、来源可信。

**检查项**：
- [ ] 漏洞公告扫描（CVE/RUSTSEC/GHSA）通过
- [ ] 无禁用 crate/包
- [ ] 无未维护依赖（或已记录忽略理由）
- [ ] 无 yanked 版本
- [ ] 来源审查（无可疑镜像源）
- [ ] 许可证扫描通过

**工具**：
- Rust: `cargo audit` + `cargo deny check`
- Node.js: `npm audit` / `pnpm audit` / `snyk`
- Python: `pip-audit` / `safety` / `bandit`
- Go: `govulncheck` / `nancy`
- 通用: `trivy` / `snyk` / `dependabot`

**通过标准**：0 未忽略的 Critical/High 漏洞，所有忽略项有明确理由和跟踪计划。

---

### 2.3 敏感信息审计

**目标**：确保代码库中不包含任何真实生产凭据。

**检查项**：
- [ ] Grep 扫描真实密码（数据库/Redis/API key/Token）
- [ ] Grep 扫描真实公网 IP
- [ ] Grep 扫描真实本机路径（如 `E:\` / `/home/user/`）
- [ ] Grep 扫描真实数据库名/表空间名
- [ ] Grep 扫描真实 SSH 私钥/证书
- [ ] 检查 `.gitignore` 是否排除 `.env` / `*.key` / `*.pem`
- [ ] 检查 Git 历史是否曾泄露凭据（需用 `git log -p | grep`）
- [ ] 验证文档示例使用占位符（`<your-password>` / `example.com`）

**工具**：
- `gitleaks` / `trufflehog` / `git-secrets`：自动扫描 Git 历史
- `grep -rE "(password|secret|key|token)\s*[:=]\s*['\"][^'\"]{8,}"`：手动扫描
- 通用正则：`[0-9]{1,3}\.[0-9]{1,3}\.[0-9]{1,3}\.[0-9]{1,3}`（IP）

**保留例外**（需明确记录）：
- 测试代码中的本地 `127.0.0.1` + 默认密码（如 `test123`）
- CI service container 初始化密码（必须固定）
- 文档示例占位符（`<your-xxx>`）

**通过标准**：0 真实生产凭据泄露。如发现，**立即修改生产环境密码** + 清除 Git 历史（`git filter-repo`）+ 重新发布。

## 三、P1 强烈建议审计（正式版前）

### 3.1 依赖供应链审计

**目标**：验证传递依赖来源可信、维护者活跃、无 typosquatting。

**检查项**：
- [ ] 列出所有传递依赖树
- [ ] 检查依赖来源（官方 registry / Git / path）
- [ ] 检查维护者活跃度（最近发布时间 < 1 年）
- [ ] 检查是否存在 typosquatting（仿冒包名）
- [ ] 检查 SLSA Level（>= L2 优先）
- [ ] 检查依赖是否被官方仓库索引

**工具**：
- Rust: `cargo supply-chain` / `cargo tree`
- Node.js: `npm ls` / `pnpm why` / `socket.dev`
- Python: `pipdeptree` / `pip-audit`
- Go: `go mod graph`
- 通用: `snyk` / `dependabot`

**通过标准**：所有直接依赖来自官方源，无 typosquatting，无活跃度异常（> 2 年未更新需评估替代方案）。

---

### 3.2 许可证合规审计

**目标**：确保所有依赖许可证与项目商业目标兼容。

**检查项**：
- [ ] 列出所有依赖许可证
- [ ] 检查 GPL/AGPL 传染性许可证（商业项目需特别关注）
- [ ] 检查 SSPL/BSL 等非 OSI 许可证
- [ ] 检查项目自身许可证与依赖许可证兼容性
- [ ] 检查 NOTICE 文件完整性（Apache 2.0 要求）

**工具**：
- Rust: `cargo deny check licenses`
- Node.js: `license-checker` / `license-compatibility-checker`
- Python: `pip-licenses`
- Go: `go-licenses`
- 通用: `fosstars`

**通过标准**：所有依赖许可证在 allow-list 内，或已获法务审批。

---

### 3.3 代码覆盖率审计

**目标**：验证测试覆盖率达标，识别未覆盖的关键路径。

**检查项**：
- [ ] 行覆盖率 ≥ 80%（核心包 ≥ 90%）
- [ ] 分支覆盖率 ≥ 70%
- [ ] 函数覆盖率 ≥ 85%
- [ ] 关键路径 100% 覆盖（事务/连接池/安全模块）
- [ ] 未覆盖代码无 Critical 逻辑
- [ ] 覆盖率趋势无下降（回归检测）

**工具**：
- Rust: `cargo tarpaulin` / `cargo llvm-cov` + Codecov/Coveralls
- Node.js: `c8` / `istanbul` / `nyc`
- Python: `coverage.py` / `pytest-cov`
- Go: `go test -cover` / `gocov`

**通过标准**：行覆盖率 ≥ 80%，关键路径 100%，无下降趋势。

---

### 3.4 API 兼容性审计

**目标**：确保公共 API 向后兼容，避免 breaking change。

**检查项**：
- [ ] semver 违规检测（minor/patch 不应引入 breaking）
- [ ] 公共 API 变更日志（CHANGELOG.md）
- [ ] deprecated 标记完整性
- [ ] breaking change 显式标注（`#[deprecated]` / `@deprecated`）
- [ ] API 文档与实现一致

**工具**：
- Rust: `cargo semver-checks`
- Node.js: `npm-package-arg` / `semver`
- Python: `pylint --disable=all --enable=deprecated`
- Go: `apidiff`
- 通用: OpenAPI diff / gRPC breaking change detector

**通过标准**：0 semver 违规，所有 breaking change 已在 CHANGELOG 显式标注。

---

### 3.5 文档完整性审计

**目标**：验证文档与代码一致，链接有效，示例可运行。

**检查项**：
- [ ] API 文档生成无警告
- [ ] 文档示例全部可运行（doctest）
- [ ] Markdown 链接全部有效（无 404）
- [ ] README 快速开始步骤可复现
- [ ] 版本号、日期、统计数据统一
- [ ] 文档与代码行为一致（无过期描述）
- [ ] 多语言文档同步（如有 i18n）

**工具**：
- `markdown-link-check`：链接有效性
- `lychee`：高性能链接检查
- `cargo test --doc`：Rust doctest
- `doctest`：Python doctest
- `markdownlint`：Markdown 规范

**通过标准**：0 失效链接，0 doctest 失败，0 过期描述。

---

### 3.6 性能回归审计

**目标**：建立性能基线，检测回归，设置性能预算。

**检查项**：
- [ ] 关键路径有基准测试
- [ ] 基准测试覆盖核心场景（CRUD/查询/事务/连接池）
- [ ] 性能基线已记录（CSV/JSON）
- [ ] 回归检测阈值（如 ±10%）
- [ ] CI 自动性能报警
- [ ] 性能预算文档化

**工具**：
- Rust: `criterion` / `divan` / `iai`
- Node.js: `benchmark.js` / `mitata`
- Python: `pytest-benchmark` / `asv`
- Go: `go test -bench` / `benchstat`

**通过标准**：关键路径 0 回归（> 10%），性能趋势稳定。

## 四、P2 中期审计（金融级前）

### 4.1 模糊测试审计

**目标**：验证异常输入下的健壮性。

**检查项**：
- [ ] 关键解析器有 fuzzing（SQL/JSON/protobuf/URL）
- [ ] Fuzzing 持续运行（CI 或 OSS-Fuzz）
- [ ] Property-Based Testing 覆盖核心不变量
- [ ] panic 恢复机制验证
- [ ] 内存安全验证（无 UAF/DF/buffer overflow）

**工具**：
- Rust: `cargo fuzz` / `proptest` / `quickcheck`
- Node.js: `fast-check` / `jsfuzz`
- Python: `hypothesis` / `atheris`
- Go: `go-fuzz` / `go test -fuzz`

**通过标准**：关键解析器 0 panic，24h fuzzing 无 crash。

---

### 4.2 可观测性审计

**目标**：验证监控、告警、追踪覆盖度。

**检查项**：
- [ ] 关键指标有 metrics（QPS/延迟/错误率/资源）
- [ ] 分布式追踪覆盖（OpenTelemetry）
- [ ] 日志级别合理（INFO/WARN/ERROR）
- [ ] SLO 告警阈值合理（燃烧率）
- [ ] Grafana dashboard 完整
- [ ] Alertmanager 告警规则配置

**工具**：
- Prometheus + Grafana + Alertmanager
- OpenTelemetry + Jaeger/Tempo
- ELK / Loki + Promtail

**通过标准**：关键指标 100% 覆盖，SLO 告警 5 分钟窗口 + 1 小时窗口双窗口配置。

---

### 4.3 灾备方案审计

**目标**：验证备份恢复流程、降级预案、混沌工程。

**检查项**：
- [ ] 备份策略文档化（全量/增量/频率）
- [ ] 恢复演练已完成（RTO/RPO 达标）
- [ ] 降级预案文档化（数据库故障/缓存故障/网络分区）
- [ ] 混沌工程覆盖（网络延迟/节点宕机/磁盘满）
- [ ] 故障注入演练记录
- [ ] 事后复盘（Post-mortem）流程

**工具**：
- Chaos Mesh / Chaos Monkey
- Litmus / Gremlin
- Velero（Kubernetes 备份）

**通过标准**：RTO ≤ 15 分钟，RPO ≤ 5 分钟，混沌工程 16 项全通过。

---

### 4.4 并发安全审计

**目标**：验证并发代码无数据竞争、死锁、锁竞争。

**检查项**：
- [ ] Send/Sync 边界正确（Rust）
- [ ] 锁粒度合理（无过大临界区）
- [ ] 无死锁风险（锁顺序一致）
- [ ] 无数据竞争（原子操作正确）
- [ ] 异步代码无阻塞（async 中无 sync I/O）
- [ ] ThreadSanitizer 检测通过

**工具**：
- Rust: `cargo miri` / ThreadSanitizer / `loom`
- Node.js: `claudia` / `doctor`
- Python: `threading` 静态分析
- Go: `go test -race` / `go vet -race`

**通过标准**：0 数据竞争，0 死锁，TSan 检测通过。

---

### 4.5 内存安全审计

**目标**：验证无内存泄漏、UAF、DF、buffer overflow。

**检查项**：
- [ ] unsafe 块审查（Rust）
- [ ] 无内存泄漏（Valgrind/ASan 检测）
- [ ] 无 use-after-free
- [ ] 无 double-free
- [ ] 无 buffer overflow
- [ ] 无未初始化内存读取
- [ ] 资源释放正确（RAII / Drop / defer）

**工具**：
- Rust: `cargo miri` / AddressSanitizer / MemorySanitizer
- C/C++: Valgrind / ASan / MSan / UBSan
- Go: `go vet` / `govet -shadow`
- Python: `tracemalloc` / `objgraph`

**通过标准**：MirI 0 报错，Valgrind 0 definitely lost，ASan 0 error。

---

### 4.6 配置审计

**目标**：验证配置完整性、最小化、feature flag 组合。

**检查项**：
- [ ] Cargo.toml/package.json/pyproject.toml 配置完整
- [ ] feature flag 组合全部可编译
- [ ] 无未使用依赖
- [ ] 无重复依赖（或已记录理由）
- [ ] 版本锁定文件完整（Cargo.lock / package-lock.json）
- [ ] 默认配置最小化（按需启用 feature）

**工具**：
- Rust: `cargo hack check --feature-powerset` / `cargo udeps`
- Node.js: `depcheck` / `npm-check`
- Python: `pip-tools` / `deptry`
- Go: `go mod tidy` / `deadcode`

**通过标准**：feature powerset 100% 编译通过，0 未使用依赖。

## 五、P3 长期审计（5.0/5 满分级）

### 5.1 可访问性审计

**目标**：验证对残障用户的友好度。

**检查项**：
- [ ] CLI 支持 `--no-color` 选项
- [ ] Web 界面 WCAG 2.1 AA 合规
- [ ] 错误消息对屏幕阅读器友好
- [ ] 键盘导航完整（无鼠标可操作）
- [ ] 颜色对比度 ≥ 4.5:1

**工具**：
- axe-core / Pa11y / Lighthouse（Web）
- `color-contrast` 检查器
- 屏幕阅读器测试（NVDA / VoiceOver）

---

### 5.2 国际化审计

**目标**：验证多语言支持完整性。

**检查项**：
- [ ] 错误消息支持 i18n
- [ ] 日志支持 i18n
- [ ] 文档多语言版本同步
- [ ] 日期/时间/数字本地化
- [ ] 字符编码 UTF-8 全覆盖

**工具**：
- `i18next` / `fluent` / `gettext`
- `crowdin` / `transifex`（翻译管理）

---

### 5.3 可维护性审计

**目标**：验证圈复杂度、耦合度、技术债务可控。

**检查项**：
- [ ] 圈复杂度 ≤ 15（函数级）
- [ ] 模块耦合度合理（无循环依赖）
- [ ] 内聚度高（单一职责）
- [ ] 技术债务有跟踪（TODO/FIXME 清单）
- [ ] 代码重复率 < 5%

**工具**：
- Rust: `cargo clippy --warn clippy::complexity` / `cargo modules`
- SonarQube / CodeClimate
- `lizard`（多语言圈复杂度）
- `jscpd`（重复代码检测）

---

### 5.4 可重复构建审计

**目标**：验证构建环境可重现。

**检查项**：
- [ ] 依赖锁定文件完整
- [ ] 构建步骤文档化
- [ ] Docker 镜像可重复构建
- [ ] Nix flake 可重复构建（可选）
- [ ] 不同机器构建结果一致（bit-for-bit reproducible）

**工具**：
- `nix build` / `nix flakes`
- `docker build --reproducible`
- `reproducible-builds` 工具链

---

### 5.5 生产案例审计

**目标**：验证有真实业务采纳，有生产事故复盘。

**检查项**：
- [ ] 至少 1 个真实生产案例
- [ ] GitHub Stars / Downloads 趋势
- [ ] 社区活跃度（Issues/PR 处理速度）
- [ ] 生产事故 Post-mortem 记录
- [ ] 用户反馈收集机制

**工具**：
- GitHub Insights / Star History
- `crates.io` / `npmjs.com` 下载统计
- 用户调研问卷

---

### 5.6 合规性审计

**目标**：验证符合数据合规要求。

**检查项**：
- [ ] GDPR 合规（如服务欧盟用户）
- [ ] 等保 2.0 合规（如服务中国境内）
- [ ] SOC 2 Type II 认证（如服务美国企业）
- [ ] 数据流图审查
- [ ] 数据保留策略文档化
- [ ] 用户数据删除流程
