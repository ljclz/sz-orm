## 二、SOC 2 Type II 审计方案

### 2.1 SOC 2 概述

**SOC 2**（Service Organization Control 2）是 AICPA（美国注册会计师协会）制定的审计标准，针对服务组织的**安全性、可用性、处理完整性、机密性、隐私性**五大信任服务原则进行审计。

- **Type I**：审计师评估**控制设计**的合理性（某一时点快照）
- **Type II**：审计师评估**控制运行有效性**（3-12 个月观察期）

**L4 金融级要求**：必须通过 SOC 2 Type II，观察期至少 6 个月。

### 2.2 SOC 2 对 ORM 项目的适用性分析

**关键问题**：SOC 2 通常针对 SaaS 服务组织，SZ-ORM 是**开源库**，不直接提供托管服务。

**两种路径**：

#### 路径 A：作为"处理完整性"组件被审计

将 SZ-ORM 嵌入一个使用 SOC 2 审计的系统中，审计师评估 SZ-ORM 的控制点：

- **处理完整性（PI）**：ORM 的 CRUD 操作是否完整、准确、及时
- **机密性（C）**：数据库凭证、查询数据是否被妥善保护
- **安全性（CC）**：SQL 注入、连接池泄漏等安全控制

#### 路径 B：作为"开发者工具"自我声明

参考 [SOC 2 for Open Source](https://openssf.org/blog/2023/02/14/soc-2-for-open-source-projects/)，建立**控制矩阵**并聘请审计师审计开源项目的开发流程：

- 代码审查流程（PR review、CODEOWNERS）
- 持续集成（CI/CD pipeline、secret scanning）
- 漏洞披露（SECURITY.md、CVE 流程）
- 发布管理（语义化版本、签名发布）

**推荐路径**：路径 B（成本更低，适用于开源项目）。

### 2.3 SOC 2 Type II 审计前置准备（6-12 个月）

#### 阶段 1：建立控制矩阵（1-2 个月）

**Common Criteria（CC1-CC9）控制点**：

| 控制域 | 控制点 | SZ-ORM 现状 | 待补强 |
|--------|--------|-------------|--------|
| CC1 控制环境 | 代码所有权、贡献者协议 | 无 CODEOWNERS | 添加 CODEOWNERS + CLA |
| CC2 沟通与信息 | 文档、CHANGELOG | 有文档 | 添加 SECURITY.md + 漏洞披露流程 |
| CC3 风险评估 | 威胁建模 | 无 | STRIDE 威胁建模文档 |
| CC4 监控 | CI/CD、依赖扫描 | 有 CI | 添加 Snyk/Dependabot |
| CC5 控制活动 | 代码审查、合并门控 | 有 PR review | 添加必审人数 ≥2 |
| CC6 逻辑与物理访问 | 仓库权限、密钥管理 | 2FA 已启用 | 添加密钥轮换策略 |
| CC7 系统运营 | 监控、事件响应 | 有 Prometheus | 添加事件响应 runbook |
| CC8 变更管理 | 发布流程、回滚 | 有语义化版本 | 添加发布签名（sigstore） |
| CC9 风险缓解 | 业务连续性 | 无 | 添加 BCP 文档 |

#### 阶段 2：控制运行观察期（6 个月）

- 每月收集控制证据（CI 日志、PR 审查记录、漏洞披露响应时间）
- 每季度执行内部审计
- 修复发现的控制缺陷

#### 阶段 3：正式审计（2-3 个月）

1. **选择审计师**：Big 4（Deloitte/PwC/EY/KPMG）或专业公司（A-LIGN/Prescient Assurance）
2. **审计费用**：开源项目通常 $30,000-$80,000（Type II）
3. **审计范围**：开源开发流程 + CI/CD + 漏洞披露
4. **审计输出**：SOC 2 Type II 报告（含审计师意见）

## 三、渗透测试方案

### 3.1 渗透测试概述

**渗透测试**（Penetration Testing）是模拟攻击者对系统进行授权攻击测试，发现可被利用的安全漏洞。

**L4 金融级要求**：必须由**第三方安全公司**执行，每年至少 1 次。

### 3.2 渗透测试范围（针对 ORM 库）

ORM 库的渗透测试与 Web 应用不同，重点是**库本身的安全性**：

#### 测试维度 1：SQL 注入（SQLi）

| 测试点 | 测试方法 | 预期结果 |
|--------|----------|----------|
| 字符串拼接 SQL | 构造恶意表名/列名 | 参数化查询应阻止注入 |
| MorphTo 关系加载 | 注入 `morph_type_value` | 应校验标识符白名单 |
| 原生 SQL 接口 | `execute_raw` 注入 | 应强制参数化 |
| 分页 SQL 构造 | 极大 OFFSET 值 | 应限制最大值 |
| LIKE 查询 | `%`/`_` 通配符注入 | 应支持转义 |

**SZ-ORM 已知风险**：反向审计发现 C-2 SQL 注入（MorphTo 拼接 `morph_type_value`）— **必须修复**。

#### 测试维度 2：资源耗尽（DoS）

| 测试点 | 测试方法 | 预期结果 |
|--------|----------|----------|
| 连接池耗尽 | 大量并发请求不释放 | 应有超时与回收 |
| 内存爆炸 | 查询超大结果集 | 应支持分页/流式 |
| 无限递归 | 关系加载自引用 | 应有最大深度限制 |
| 死锁 | 并发事务竞争 | 应有死锁检测 |

#### 测试维度 3：数据完整性

| 测试点 | 测试方法 | 预期结果 |
|--------|----------|----------|
| 事务回滚 | 中断后查询一致性 | ACID 保证 |
| 并发更新 | 乐观锁冲突 | 应有 version 控制 |
| 浮点精度 | 金额计算精度 | 应使用 Decimal |
| 时区处理 | 跨时区时间查询 | 应统一 UTC |

#### 测试维度 4：凭证与机密

| 测试点 | 测试方法 | 预期结果 |
|--------|----------|----------|
| 连接字符串泄漏 | 日志/错误信息 | 应脱敏 |
| 内存中的凭证 | core dump 分析 | 应及时清零 |
| 测试凭证残留 | 代码扫描 | 应使用占位符 |

**SZ-ORM 已知风险**：已修复 31 处真实密码/IP 泄漏，但需第三方验证。

### 3.3 渗透测试方法论

#### 阶段 1：白盒测试（代码审计）

**工具**：
- **Semgrep**：静态分析，自定义 Rust 规则
- **CodeQL**：GitHub 语义分析
- **cargo-audit** + **cargo-deny**：依赖漏洞
- **cargo-fuzz**：模糊测试（已配置）

**检查项**：
- 所有 `format!`/`format_args!` 拼接 SQL 的位置
- 所有 `unsafe` 块（应为 0）
- 所有 `.unwrap()`/`.expect()`（应有上下文）
- 所有 `as` 类型转换（检查溢出）

#### 阶段 2：黑盒测试（运行时测试）

**工具**：
- **sqlmap**：自动化 SQL 注入
- **cargo-fuzz** + **proptest**：模糊输入
- **自定义压测脚本**：模拟 DoS 场景

**测试场景**：
1. 启动真实 PostgreSQL/MySQL
2. 通过 SZ-ORM API 执行各种查询
3. 注入恶意输入（特殊字符、超长字符串、Unicode）
4. 监控数据库日志、ORM 日志、内存/CPU

#### 阶段 3：第三方独立测试

**推荐供应商**（按价格/专业度）：
- **Trail of Bits**：Rust 生态专家，$50,000-$150,000
- **NCC Group**：大型安全公司，$40,000-$120,000
- **Cure53**：德国公司，€30,000-€90,000
- **GitHub Security Lab**：开源项目免费申请

**交付物**：
- 渗透测试报告（含 CVSS 评分）
- 漏洞复现步骤
- 修复建议
- 复测报告（修复后）

**综合就绪度**：约 38%，预计需要 2-3 个月补强后可启动第三方测试。
